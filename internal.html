Technical Deep-Dive

CyberDarkNet Crypto Internals

How end-to-end encryption works, why the server cannot decrypt your messages, and where every key lives.

🔐 Zero-Trust Architecture ⚡ Signal Protocol 🦀 Rust + ring + aes-gcm
TL;DR

The server is a blind courier. It stores ciphertext it cannot read, routes packets it cannot inspect, and verifies signatures it did not produce. All keys that matter live only on client devices. Here is exactly how.

🤔 The Problem This Solves

Traditional chat servers are trusted third parties. Even with HTTPS, the server decrypts your message, stores it in plaintext, and re-encrypts it for the recipient. If the server is compromised — by a hacker, a subpoena, or a rogue admin — every message is exposed.

CyberDarkNet inverts this. The server only ever sees:

  • Ciphertext blobs (BYTEA columns it cannot interpret)
  • Public keys (by definition, meant to be public)
  • Message metadata (sender device ID, conversation ID, timestamps)

The private keys never leave the client. The server does not have them. Even if you hand an attacker root access to the database, they get encrypted noise.

🔑 Key Types in Play

Two fundamentally different key types are used, each for a different job.

Ed25519 Keys — Identity and Signing

Ed25519 is a digital signature algorithm. A private key signs data; anyone with the corresponding public key can verify the signature. It proves who created something, not who can read something.

Used for:

  • Device identity: each device registers an identity_key (Ed25519 public key)
  • Signed prekeys: the device signs its X25519 prekey so recipients know it wasn't swapped by the server
  • Federation: servers sign their handshake and relay packets to prevent impersonation
src/crypto/keys.rs
pub struct Ed25519KeyPair {
    pub signing_key:   SigningKey,    // private — never leaves device
    pub verifying_key: VerifyingKey, // public  — uploaded to server
}

The server stores Ed25519 public keys but cannot produce valid signatures with them. A compromised server cannot forge device identity.

X25519 Keys — Key Agreement (Diffie-Hellman)

X25519 is an elliptic-curve Diffie-Hellman function. Two parties each have a private/public keypair. They exchange public keys and independently compute the same shared secret, without the shared secret ever being transmitted.

Used for:

  • identity_key (X25519): long-term device DH key
  • signed_prekey (X25519): medium-term DH key, rotated periodically
  • one_time_prekeys (X25519): single-use DH keys, consumed one per session
src/crypto/keys.rs
impl X25519KeyPair {
    pub fn diffie_hellman(&self, their_pub: &[u8; 32]) -> [u8; 32] {
        x25519(self.secret, *their_pub) // both sides compute same result
    }
}

The server stores only the public halves. The private halves stay on the device. Without private keys, DH outputs cannot be computed.

📋 Phase 1: Device Registration

When a device first connects, it uploads a prekey bundle — a collection of public keys:

HTTP
POST /api/v1/devices/register
{
  "identity_key":            "<Ed25519 public, base64>",
  "signed_prekey":           "<X25519 public, base64>",
  "signed_prekey_signature": "<Ed25519 sig of signed_prekey, base64>",
  "one_time_prekeys": [
    { "key_id": 1, "public_key": "<X25519 public, base64>" },
    { "key_id": 2, "public_key": "<X25519 public, base64>" }
  ]
}

The server performs one critical check before storing:

src/api/devices.rs
verify_signed_prekey(
    &req.identity_key,           // Ed25519 verifying key
    &req.signed_prekey,          // X25519 public key (as bytes)
    &req.signed_prekey_signature // Ed25519 sig of those bytes
)

This confirms the signed_prekey was produced by the same device that owns the identity_key. The server cannot fake this relationship — it does not have the Ed25519 private key needed to produce the signature.

After this, public keys are stored in PostgreSQL and the server has nothing useful for decryption — it never had the private keys to begin with.

🤝 Phase 2: Session Initiation — X3DH

Before Alice can send Bob a message, they need a shared secret. They have never talked before. The server cannot learn the secret. This is solved by Extended Triple Diffie-Hellman (X3DH), the same algorithm Signal uses.

Alice fetches Bob's prekey bundle

HTTP
GET /api/v1/devices/{bob_device_id}/prekey-bundle
→ {
    identity_key, signed_prekey, signed_prekey_signature,
    one_time_prekey  // consumed — server marks it used, can't replay
  }

Alice runs X3DH locally

src/crypto/x3dh.rs
pub fn initiate(
    alice_identity_priv: [u8; 32],   // Alice's private key — never sent
    alice_ephemeral_priv: [u8; 32],  // fresh random key — never sent
    bob_bundle: &PrekeyBundle,       // what server returned
) -> ([u8; 32], X3DHMessage)         // (shared_secret, what to send Bob)

Four DH operations are combined. Each one adds a layer of security:

Operation Alice's key Bob's key Security property
DH1Identity (private)Signed prekey (public)Authenticates Alice
DH2Ephemeral (private)Identity (public)Authenticates Bob
DH3Ephemeral (private)Signed prekey (public)Forward secrecy
DH4Ephemeral (private)One-time prekey (public)Deniability + extra secrecy
src/crypto/x3dh.rs
let dh1 = x25519(alice_identity_priv,  bob_spk);
let dh2 = x25519(alice_ephemeral_priv, bob_ik);
let dh3 = x25519(alice_ephemeral_priv, bob_spk);
let dh4 = x25519(alice_ephemeral_priv, bob_opk); // if one-time prekey exists

These four outputs are concatenated and fed into HKDF-SHA256:

src/crypto/kdf.rs
pub fn x3dh_kdf(dh_outputs: &[&[u8; 32]]) -> [u8; 32] {
    let mut ikm = F.to_vec(); // 32×0xFF padding prefix (per X3DH spec)
    for dh in dh_outputs { ikm.extend_from_slice(*dh); }
    hkdf_sha256(&[0u8; 32], &ikm, b"cyberdarknet-x3dh-v1")
}

Alice now has a 32-byte shared_secret. She sends Bob her public ephemeral key and identity key (the X3DHMessage). The server relays this header alongside the ciphertext.

Bob reconstructs the same secret

Bob runs the DH operations in reverse order but arrives at the same 32-byte secret. The server watched the key exchange but cannot compute any of the DH outputs — it only ever saw public keys.

Why the server is blind here: Each DH output requires one private key. The server has zero private keys in these exchanges — all private keys were generated locally on devices and never transmitted.

🔄 Phase 3: Ongoing Messages — Double Ratchet

X3DH establishes one shared secret. But using the same key for every message is dangerous: if one key leaks, all past and future messages are exposed. The Double Ratchet solves this by deriving a new encryption key for every single message.

Symmetric Ratchet — chain key → message key

src/crypto/kdf.rs
pub fn kdf_ck(chain_key: &[u8; 32]) -> ([u8; 32], [u8; 32]) {
    let msg_key = hmac_sha256(chain_key, &[0x01]); // key for this message
    let new_ck  = hmac_sha256(chain_key, &[0x02]); // advance the chain
    (new_ck, msg_key)
}

Each call advances the chain and produces a one-use message key. Knowing message key #5 does not let you derive message key #4 (HMAC is one-way). This gives forward secrecy within a chain.

DH Ratchet — root key refresh

src/crypto/kdf.rs
pub fn kdf_rk(root_key: &[u8; 32], dh_output: &[u8; 32]) -> ([u8; 32], [u8; 32]) {
    let derived = hkdf_sha256(root_key, dh_output, b"cyberdarknet-ratchet-v1");
    let new_rk = hkdf_sha256(&derived, &[0x01], RATCHET_INFO);
    let new_ck = hkdf_sha256(&derived, &[0x02], RATCHET_INFO);
    (new_rk, new_ck)
}

Every time the conversation direction flips (Alice sends, then Bob replies), a new DH exchange happens inline. This rotates the root key. An attacker who steals the root key at time T cannot decrypt messages from before T (forward secrecy) and cannot predict keys after T is rotated (break-in recovery).

Message Encryption — AES-256-GCM

src/crypto/aes.rs
pub fn encrypt(key: &[u8; 32], plaintext: &[u8]) -> Result<Vec<u8>, AppError> {
    // random 12-byte nonce, unique per message
    OsRng.fill_bytes(&mut nonce_bytes);
    // output layout: nonce(12 bytes) || ciphertext+tag
    out.extend_from_slice(&nonce_bytes);
    out.extend_from_slice(&ciphertext);
}

The server receives and stores only this nonce || ciphertext blob. The 16-byte GCM authentication tag is embedded in the ciphertext — if anyone tampers with the blob, decryption fails with an authentication error. The server stores it but has no message key to run decrypt() with.

💥 Compromise Scenarios

Each scenario assumes progressively stronger attacker capabilities. The architecture is designed so that even total server compromise does not expose plaintext messages.

1 Attacker reads the entire PostgreSQL database Low impact
What the attacker gets
  • messages.ciphertext — raw BYTEA blobs, one per message. No column for plaintext exists in the schema.
  • devices.identity_key, devices.signed_prekey, one_time_prekeys.public_key — all public halves only. The DB schema has no private_key column. Private keys were generated on the device and never transmitted.
  • users.password_hash — Argon2id hash with random salt. Not the password itself.
  • Conversation metadata — who talked to whom, when, message counts. Content is absent.
What the attacker cannot do
  • Decrypt any message. Every DH operation (X3DH + Double Ratchet) requires a private key. None are in the DB. The AES-256-GCM message keys are derived on-device and never stored anywhere.
  • Reverse password hashes. Argon2id with default parameters (memory=64MB, iterations=3) makes brute-force cost roughly $50,000+ per password on modern hardware. Weak passwords remain vulnerable — strong passwords are infeasible.
  • Reconstruct ratchet state. Ratchet chain keys advance with every message and are only held in device memory. They are not persisted to the server.
Result

Attacker holds encrypted noise and public keys. Without device private keys — which lived only in client memory and were never uploaded — the ciphertext is mathematically opaque. The entire database dump is worthless for reading messages.

Precautions for operators
Encrypt the database volume at rest (LUKS / cloud provider disk encryption). This protects against physical disk theft even before an attacker can query tables.

Rotate TOKEN_SECRET immediately after detecting a DB breach — all existing sessions become invalid, forcing re-login.

Alert on unusual query patterns — a full table scan of messages or devices by an unknown connection is a strong breach signal.
2 Attacker owns the running server process (root / RCE) Medium impact
What the attacker gets
  • Everything from Scenario 1.
  • TOKEN_SECRET (env var) — can mint arbitrary valid JWTs for any user ID. Can impersonate any account going forward.
  • SERVER_PRIVATE_KEY (env var) — can sign fake federation packets, impersonate this server to peers. Can inject messages into federated conversations as if they came from this server.
  • Live WebSocket connections — can inspect the envelope of messages in flight (device IDs, conversation IDs, timestamps). The payload is still ciphertext.
  • Future traffic — can serve malicious prekey bundles to clients that connect after the compromise.
What the attacker cannot do
  • Decrypt any stored message. The crypto modules (src/crypto/ratchet.rs, aes.rs, x3dh.rs) are reference implementations only — they exist for client libraries to copy, not for server use. No API handler calls decrypt(). There is no code path on the server that could decrypt even if you hand it a key.
  • Decrypt messages in flight. Messages arrive at the server already encrypted. The server's job is to store and forward bytes, not parse them. Even with process-level access, what you see in the WebSocket handler is ciphertext: BYTEA.
  • Impersonate a specific device to a contact who has verified keys. Forged JWTs use a valid user UUID but can't prove device identity to a contact who has already fingerprint-verified that device's Ed25519 key.
Result

Serious incident — attacker controls the transport layer going forward. They can read who talks to whom, impersonate users in new conversations, and poison new sessions with fake prekey bundles. But every message sent before the compromise remains encrypted and unreadable. Past history is protected by forward secrecy.

Real danger: future session poisoning
After process compromise, the attacker can serve malicious prekey bundles for Bob's device — bundles with keys they control. If Alice starts a new conversation after the compromise, she would encrypt to the attacker's keys, not Bob's. This is a classic key substitution attack.

This is only preventable by out-of-band key verification (comparing key fingerprints over a separate channel — like Signal's "safety numbers"). CyberDarkNet does not yet implement this UX.
Precautions for operators
Use a secrets manager (HashiCorp Vault, AWS Secrets Manager) instead of env vars for TOKEN_SECRET and SERVER_PRIVATE_KEY. Secrets injected at runtime via a sidecar are harder to exfiltrate than env vars visible in /proc/self/environ.

Run the server as a non-root user with minimal filesystem permissions. RCE as a restricted user limits lateral movement.

Rotate SERVER_PRIVATE_KEY after any suspected compromise. Peer servers will reject packets signed with the old key once you re-handshake.

Implement key transparency / append-only prekey log — clients should be alerted when a device's prekey bundle changes unexpectedly (future feature).
3 Network-level man-in-the-middle (between client and server) Medium impact — with important caveats
What the attacker gets
  • Plaintext HTTP traffic if TLS is not used — all API calls, including prekey bundles and ciphertext uploads.
  • With TLS: only timing and packet sizes. Content is opaque.
  • The opportunity to swap prekey bundles in transit — if TLS is stripped or the MITM holds a valid cert (e.g., corporate proxy, rogue CA).
What the attacker cannot do
  • Swap only the signed_prekey without detection. The signed_prekey_signature is an Ed25519 signature of the prekey bytes, made with Bob's long-term Ed25519 private key. Replacing the prekey with their own X25519 public key produces an invalid signature. A client that verifies this before running X3DH will reject it.
  • Forge federation packets — all inter-server packets carry a timestamp, nonce, and Ed25519 signature. Replay and substitution are detected.
The honest caveat — two real attack vectors remain
Attack A — Full identity substitution: A MITM who swaps both identity_key and signed_prekey with their own keypair produces a perfectly valid bundle (their signature on their key). Alice has no way to detect this from the bundle alone. She encrypts to the attacker, who relays to Bob with their own session — classic man-in-the-middle. This is only defeated by out-of-band fingerprint comparison (comparing the hex of Bob's identity_key over a phone call). CyberDarkNet does not yet surface key fingerprints in the UI.

Attack B — Client does not verify the signature: The server verifies signed_prekey_signature at device registration (src/api/auth.rs:33-37). But the x3dh::initiate() reference implementation marks this as "verified separately" — meaning the client library is responsible for calling verify_signed_prekey() before using the bundle. A client that skips this step is vulnerable to a prekey swap even without full identity substitution.
Result

With TLS and proper client-side signature verification: MITM cannot read messages. Without TLS or without client-side verification: MITM can intercept new sessions. Existing sessions (already established via X3DH) are unaffected — the attacker cannot retroactively decrypt them without the original private keys.

Precautions for operators and client developers
Always use HTTPS. TLS is the first and most important defense. Without it, every API call is readable on the wire. Use Let's Encrypt or a reverse proxy (nginx/Caddy) — never run the backend on plain HTTP in production.

Client libraries must verify signed_prekey_signature before calling x3dh::initiate(). The server already verified it at registration, but transport-layer tampering can swap the served value. Client re-verification closes this gap.

Implement key fingerprints in the UI (future feature). Each device's identity_key displayed as a short hex digest or emoji sequence, comparable out-of-band. This is the only defense against a full identity substitution attack — the same mechanism Signal calls "safety numbers."

Pin the server certificate in the mobile/desktop client. Prevents attacks using rogue CA certificates (corporate proxies, nation-state CAs).
4 Attacker compromises a client device (phone / laptop theft, malware) High impact — no cryptographic defense
What the attacker gets
  • Everything. Device memory contains the Ed25519 private key, all X25519 private keys, the current ratchet state, and all decrypted message history cached locally.
  • Active sessions — can send messages as that device, read all incoming messages.
  • Identity file — if the identity JSON is accessible on the device filesystem, they can log in from a new device using the stored user_id + device_id (they still need the password).
What the attacker cannot do (limited)
  • Decrypt messages on other devices in the same conversation — each device has its own ratchet state. Compromising Alice's phone does not expose Bob's device.
  • Access messages before the ratchet window — the Double Ratchet's skipped-message cache has a limit (MAX_SKIP = 1000). Very old messages that have been deleted from the local cache are gone.
Result

Device compromise is the hardest threat to defend against cryptographically — the keys are supposed to be on the device. The server architecture cannot help here. Defense is at the device OS level.

Precautions
Device PIN / biometric lock. Physical access without unlock should not expose keys or app data.

Store private keys in the platform secure enclave where possible (iOS Secure Enclave, Android Keystore, TPM on desktop). Keys stored there cannot be exported even with root access on many platforms.

Password-encrypt the identity file before saving to disk. The current identity file is plaintext JSON — device_id and user_id are in it, but the private keys are not (they stay in memory). Adding AES-GCM encryption of the file with a device password would prevent trivial exfiltration.

Remote device revocation. The server supports DELETE /api/v1/devices/{id}. Use it immediately on suspected device compromise — new sessions from that device will fail, and peer devices will stop accepting its prekey bundles.

🚫 Hard Invariants in Code

⛔
No decrypt endpoint ever.

The server stores only BYTEA. There is no API surface to request plaintext. Adding one would require uploading private keys first — a client-side change that would be visible and auditable.

⛔
Never log ciphertext, keys, or tokens.

Log files contain only IDs, timestamps, and routing metrics. No key material ever reaches disk outside the database's encrypted BYTEA columns.

⛔
Argon2id only for passwords.

Memory-hard, GPU-resistant. bcrypt and SHA-based password hashing are explicitly banned. The password only proves login — it is not involved in encryption.

⛔
sender_device_id is NULL for federated messages.

The server cannot attribute federated plaintext to a specific sender it relayed for. Even routing metadata is minimized.

🔒 The Server's One Cryptographic Power

The server does have one keypair: an Ed25519 key used exclusively for federation packet signing.

src/main.rs
server_keys: Arc<Ed25519KeyPair> // loaded from SERVER_PRIVATE_KEY env var

This key signs outbound federation packets so peer servers can verify the packet actually came from this server:

canonical_bytes = packet_id | source | destination | timestamp | nonce | payload_json
signature = Ed25519.sign(server_private_key, canonical_bytes)

This proves which server sent a packet. It says nothing about message content — the messages inside packets are already ciphertext the signing server also cannot read. Authenticity without visibility.
